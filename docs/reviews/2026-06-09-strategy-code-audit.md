## 2. 总体结论

当前代码可以运行，大部分测试能够通过，但存在多项会影响策略结果真实性的高严重度问题。

最核心的问题是：

- `RR1` 由代码直接构造成固定的 `2.0`，盈亏比过滤基本失效。
- 大盘环境过滤实际读取的是平安银行，而不是上证指数。
- 止损小于正常 ATR 波动时仍可能输出“可低吸”。
- “接近 Pivot”没有距离下限，远低于 Pivot 的价格也可能被判定为“突破确认”。
- 实盘扫描与历史回测未使用同一套策略口径。
- 三源并发选择不会遵守配置的优先顺序和备用重试次数。
- 不同数据源可能混用前复权和未复权价格。

因此，当前扫描结果、风险收益指标和回测报告不能直接作为策略有效性的可靠证据。

## 4. 详细问题分析

### BUG-001：盈亏比过滤基本失效

#### 问题现象

多数风险为正的候选都会得到 `RR1 = 2.0`，配置中的最小盈亏比门槛无法有效淘汰真实压力位过近的候选。

#### 涉及模块

- `analyzer/key_prices.py:71`
- `analyzer/risk_reward.py:77`
- `analyzer/decision.py:135-140`

#### 可能原因

`calculate_key_prices()` 直接将第一目标价设置为：

```python
target_1 = current_price + 2 * risk
```

随后风险收益模块再次计算：

```python
rr1 = (target_1 - current_price) / risk
```

只要风险为正，计算结果天然等于 `2.0`。

#### 触发条件

- `current_price > stop_loss`
- 未发生目标价计算异常

#### 影响

- `min_rr1: 2` 基本失去筛选能力。
- `WAIT_RR` 很难由真实压力位触发。
- 候选可能在上方压力空间不足时仍显示盈亏比合格。
- 与“RR1 必须基于真实压力位”的策略优化文档冲突。

#### 修复建议

1. `target_1` 应优先使用前高、杯沿、Pivot、平台上沿或近期明确压力位。
2. 2R 仅作为理论目标，不得直接作为决策使用的真实 `target_1`。
3. 如果最近压力位导致 `RR1 < min_rr1`，应返回 `WAIT_RR`。
4. `target_2` 可以继续使用形态量度目标或 3R。

#### 验证方式

1. 构造当前价为 100、止损为 95、最近压力位为 106 的数据。
2. 计算结果应为 `RR1 = 1.2`。
3. 最终决策应为 `WAIT_RR`，不能输出 `BUY_LOW` 或 `WATCH_BREAKOUT`。

### BUG-002：大盘环境过滤使用了错误标的

#### 问题现象

系统名义上使用上证指数评估市场环境，实际请求的却是平安银行。

#### 涉及模块

- `scanner/index_source.py:9-20`
- `scanner/sina_source.py:22-25`
- `analyzer/decision.py:112`

#### 可能原因

`index_source.py` 默认指数代码为 `000001`，并复用个股新浪拉取函数。个股函数将所有非 `6` 开头代码映射为：

```python
sz000001
```

该代码对应平安银行，而不是上证指数。

#### 触发条件

每次扫描、重新评估或详情分析调用默认市场指数拉取时都会触发。

#### 影响

平安银行走势较差时，系统可能将整个市场判定为“较差”，并直接拒绝全部候选。

#### 修复建议

1. 为指数实现独立的符号映射和数据源函数。
2. 明确配置使用哪个宽基指数。
3. 不要复用仅支持 A 股个股代码映射的函数。
4. 指数获取失败时保留“默认一般”行为，但必须记录明确日志。

#### 验证方式

1. Mock 新浪请求参数。
2. 调用 `fetch_market_index_daily()`。
3. 断言请求符号是正确指数符号，而不是 `sz000001`。

### BUG-003：止损过近仍可能输出“可低吸”

#### 问题现象

当止损距离小于 `ATR14 × 1.2` 时，系统只显示警告，仍可能输出“可低吸”和较高仓位建议。

#### 涉及模块

- `analyzer/risk_reward.py:82-85`
- `analyzer/decision.py`
- `analyzer/dry_stable.py`

#### 可能原因

ATR 止损校验只向 `warnings` 添加文本，没有改变 `can_buy`、风险等级或最终决策。最终决策也没有检查 `risk_reward.can_buy`。

#### 触发条件

- 风险百分比小于 `ATR14 × atr_stop_multiplier`
- 其他量干、价稳、形态和入场区条件均满足

#### 影响

止损可能位于正常价格波动范围内，容易被正常波动触发，实际风险与系统显示不一致。

#### 修复建议

1. 增加明确的 `stop_too_close` 布尔字段或风险原因。
2. 止损过近时至少返回 `WAIT_ENTRY`。
3. 禁止止损过近的候选进入 `BUY_LOW`。
4. 最终决策应检查风险收益模块的结构化结果，而不是只读取警告文字。

#### 验证方式

1. 构造 `risk_percent=2%`、`ATR14=3%`、倍率为 `1.2` 的数据。
2. 其他条件全部满足。
3. 最终结果不得为 `BUY_LOW`。

### BUG-004：远低于 Pivot 也会判定为突破确认

#### 问题现象

当前价远低于 Pivot 时，系统仍将 `near_pivot` 判定为真，并可能输出“突破确认”。

#### 涉及模块

- `analyzer/decision.py:99-100`
- `analyzer/decision.py:166-183`

#### 可能原因

当前实现只判断：

```python
current <= pivot * 1.05
```

没有设置距离 Pivot 的下限。

#### 触发条件

任何低于 `pivot × 1.05` 的价格，包括远低于 Pivot 的价格。

#### 影响

仍处于结构下方、尚未接近突破位置的股票可能显示为“突破确认”。

#### 修复建议

1. 使用双边距离判断，例如：

```python
abs(current / pivot - 1) <= near_pivot_pct
```

2. 分开处理 Pivot 下方观察、有效突破和突破后追高。
3. 当前价远低于 Pivot 且不在低吸区时，应返回 `WAIT_ENTRY` 或拒绝状态。

#### 验证方式

1. 设置 `current=60`、`pivot=100`。
2. 断言 `near_pivot=False`。
3. 最终结果不得为 `WATCH_BREAKOUT`。

### BUG-005：历史回测没有复刻线上策略

#### 问题现象

`scanner/backtester.py` 的回测结果无法真实反映线上扫描策略的表现。

#### 涉及模块

- `scanner/backtester.py:181`
- `scanner/backtester.py:188`
- `scanner/backtester.py:253-261`
- `scanner/strategy_engine.py`

#### 可能原因

历史回测存在以下差异：

- 使用基础 `score_cup_handle()`，线上使用 `score_cup_handle_advanced()`。
- 调用干稳分析时未传入实际配置。
- 未传入市场数据。
- 不支持 VCP-only 候选。
- 使用 `breakout_price × 0.95` 作为止损，而不是策略实际止损价。
- 假突破口径也与实际策略失效规则不同。

#### 触发条件

运行 `scanner.backtester.run_backtest()` 时必然存在。

#### 影响

- 回测命中率、止损率和参数建议不代表线上策略。
- 可能根据错误回测结论调高仓位或放宽阈值。

#### 修复建议

1. 历史回测统一调用 `CupHandleStrategyEngine`。
2. 使用线上相同配置、候选规则、实际止损价和决策状态。
3. 记录每个候选的真实 `verdict_key`、`entry_zone`、`stop_loss` 和目标价。
4. VCP-only 和杯柄候选必须使用相同回测入口。

#### 验证方式

对同一历史截止日期，分别调用线上策略引擎和回测入口，断言候选状态、评分、止损价和决策完全一致。

### BUG-006：单股回测无法识别 VCP-only 候选

#### 问题现象

全市场扫描可以补充 VCP-only 候选，但单股回测无法回放同一类机会。

#### 涉及模块

- `scanner/strategy_engine.py:106-116`
- `scanner/single_stock_backtest.py:353-356`
- `scanner/engine.py:235-241`

#### 可能原因

策略引擎在杯柄未识别时直接返回，未运行干稳和 VCP 分析。单股回测随后尝试从 `evaluation.dry_stable` 判断 VCP，但该字段必然为空。

全市场扫描通过引擎外的额外补位逻辑支持 VCP，造成入口不一致。

#### 触发条件

输入数据存在成熟 VCP，但不存在有效杯柄。

#### 影响

- 扫描和单股回测结果不一致。
- 用户无法验证扫描发现的 VCP-only 信号。

#### 修复建议

将 VCP-only 分析和候选提升统一放入策略引擎。扫描、重新评估和所有回测只调用统一入口。

#### 验证方式

使用现有 `_make_vcp_data()`：

1. 策略引擎应返回 VCP 分析结果。
2. 单股回测应记录同一 VCP。
3. 全市场扫描和单股回测的 `verdict_key` 应一致。

---

### BUG-007：三源并发拉取不遵守配置优先顺序和备用重试策略

#### 问题现象

扫描配置中的三源优先顺序和主备重试次数无法可靠控制实际拉取行为。

#### 涉及模块

- `scanner/engine.py:556-626`
- `scanner/data_source.py:40-49`
- `config.yaml`

#### 可能原因

有 `DataSourceManager` 时，拉取逻辑使用 `try_acquire_any()` 随机选择管理器中任意空闲源，而不是从当前配置链中选择尚未尝试的源。

同时：

- `fallback_attempts` 参数没有用于备用源调用。
- 已失败源可能被重复选择。
- `FetchResult` 失败时丢失各源的真实尝试次数和错误。

#### 触发条件

全市场扫描的并发拉取路径。

#### 影响

- 无法保证 `baidu → sina → tencent` 的配置优先顺序。
- 备用源实际使用了主源的 `retry_attempts`，配置的 `fallback_attempts` 不生效。
- 某些可用源可能始终未被尝试。
- 任务失败详情不能准确反映真实拉取过程。

#### 修复建议

1. 明确产品语义：如果配置顺序代表优先级，则必须优先尝试链首源；只有失败或锁忙后才回退。
2. 优先尝试尚未尝试的三源，避免重复选择失败源。
3. 主源使用 `retry_attempts`，备用源使用 `fallback_attempts`。
4. 记录完整的每源尝试摘要。

#### 验证方式

- 配置为 `["baidu", "sina", "tencent"]` 时，按定义顺序选择和回退。
- 百度失败后应尝试新浪，新浪失败后应尝试腾讯。
- 备用源应使用 `fallback_attempts`。
- 一个源失败后必须尝试其他未尝试源。

### BUG-008：多源价格复权口径不一致

#### 问题现象

配置声明使用复权数据，但多个数据源没有统一执行该配置。

#### 涉及模块

- `config.yaml:18`
- `scanner/tencent_source.py:39`
- `scanner/sina_source.py`
- `scanner/baidu_source.py`
- `scanner/engine.py`

#### 可能原因

`data.use_fq: true` 未被代码读取。腾讯源明确请求 `qfq`，其他数据源没有明确统一复权处理。

#### 触发条件

扫描从不同数据源获取不同股票，或同一股票缓存由不同源更新时。

#### 影响

- 除权除息位置可能形成虚假跳空。
- 杯柄、VCP、ATR、支撑、止损和收益率可能计算错误。
- 缓存按日期合并时可能混入不同复权口径的数据。

#### 修复建议

1. 明确所有参与策略扫描的数据源必须输出前复权 OHLC。
2. 将 `use_fq` 接入所有源，或移除假配置并强制统一口径。
3. 缓存记录或校验复权类型，避免混合合并。
4. 无法保证复权的数据源不应参与策略扫描。

#### 验证方式

对同一只发生过除权的股票，比较所有启用源在同日期附近的价格连续性和比例，确保不存在口径混合。
