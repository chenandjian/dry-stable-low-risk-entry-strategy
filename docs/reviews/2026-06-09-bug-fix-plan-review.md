## 4. 必须修订的问题

### PLAN-001：历史回测必须防止未来数据泄漏

#### 原方案问题

原方案要求历史回测使用实际 `market_data`，但没有说明每个回测窗口只能使用截止当日已知的指数数据。

如果所有历史窗口都使用完整市场数据，过去日期的决策可能读取未来大盘走势，导致回测结果失真。

#### 必须修改为

回测每个日期时，必须将个股数据和市场数据同时截断：

```python
detect_date = window_data[-1]["date"]
market_window = [
    row for row in market_data
    if row["date"] <= detect_date
]

evaluation = strategy_engine.evaluate_at(
    window_data,
    code=code,
    name=name,
    market_data=market_window,
)
```

不得向历史窗口传入检测日期之后的数据。

#### 必须增加测试

1. 创建一段历史市场数据。
2. 在检测日期之后添加指数暴跌。
3. 断言检测日期当天的策略结论不受未来暴跌影响。
4. 断言回测和线上策略在相同截止日期得到相同市场状态。

### PLAN-002：RR1 不能只使用 Pivot

#### 原方案问题

原方案规定：

```text
如果 pivot > current_price，则 target_1 = pivot；
没有 pivot 时回退到 2R。
```

该设计仍不完整：

- 当前价已经突破 Pivot 时，Pivot 不再是上方压力位。
- 使用低于当前价的 Pivot 会产生负收益和负 RR1。
- 回退到 2R 会重新制造 `RR1 = 2.0` 的原问题。
- Pivot 不一定是距离当前价最近的真实压力位。

#### 必须修改为

区分“真实压力目标”和“理论目标”：

```text
target_1_real:
    当前价上方最近的真实压力位

target_2_theoretical:
    2R / 3R / 形态量度目标
```

真实压力位候选优先级建议：

1. 当前价上方最近的历史前高
2. 杯沿或尚未突破的 Pivot
3. 最近平台上沿
4. 近期明显压力位

决策使用：

```text
RR1 = (target_1_real - current_price) / risk
```

如果找不到当前价上方的真实压力位：

- 不得使用 2R 伪造真实 RR1。
- 应将 RR1 标记为未知或不足。
- 不得输出 `BUY_LOW`。

#### 必须增加测试

- 当前价低于 Pivot，Pivot 为最近压力位。
- 当前价已经突破 Pivot，必须寻找更高压力位。
- 当前价上方没有可识别压力位，不能用 2R 通过准入。
- 最近压力位导致 `RR1 < 2` 时返回 `WAIT_RR`。

### PLAN-003：ATR 止损过近必须使用结构化状态

#### 原方案问题

原方案计划在 ATR 止损过近时设置：

```python
can_buy = False
risk_level = "高风险"
```

但当前 `calculate_risk_reward()` 后续仍会重新计算风险等级，并可能将 `can_buy` 改回 `True`。

此外，`can_buy=False` 同时也用于中风险观察状态，不能单独表示“止损过近”。

#### 必须修改为

在 `RiskRewardResult` 增加明确的结构化状态，例如：

```python
stop_too_close: bool = False
```

ATR 检查触发时：

```python
r.stop_too_close = True
```

并将其纳入风险等级和最终决策：

```python
all_good = (
    not r.stop_too_close
    and ...
)
```

最终决策遇到 `stop_too_close=True` 时：

- 返回 `WAIT_ENTRY`
- 禁止 `BUY_LOW`
- 保留明确 warning

#### 必须增加测试

- ATR 止损过近时 `stop_too_close=True`。
- 即使其他条件全部满足，也不能输出 `BUY_LOW`。
- 最终状态明确为 `WAIT_ENTRY`。
- 仓位建议不能是低风险仓位。

### PLAN-004：VCP-only 统一入口必须补齐完整数据合同

#### 原方案问题

原方案仅提出取消 `StrategyEngine.evaluate_at()` 在杯柄未识别时的提前返回。

但当前引擎、候选规则和序列化主要围绕 `CupHandleResult` 设计。仅取消提前返回仍无法保证 VCP-only 正常工作：

- VCP-only 的 `result.score` 默认是 0。
- `_candidate_rules()` 仍检查 0-100 杯柄评分门槛。
- 杯柄日期字段为空。
- 单股回测的 `patternId` 可能使用空杯柄字段。
- 多个 VCP 可能被错误去重。

#### 必须修改为

方案必须明确以下设计：

1. VCP-only 如何生成 0-100 候选排序分。
2. `_candidate_rules()` 如何分别判断杯柄和 VCP-only。
3. VCP 的结构数据如何序列化。
4. VCP 的唯一标识如何生成。
5. 扫描、重新评估、单股回测和历史回测如何读取统一结果。

建议策略引擎输出统一形态结构，而不是继续让所有调用方自行补位：

```text
pattern_kind: cup_handle | vcp
pattern_score_20
candidate_score_100
pattern_identity
pattern_details
```

#### 必须增加测试

- VCP-only 能通过统一策略引擎成为候选。
- 全市场扫描和单股回测得到相同 VCP 结论。
- 历史回测能记录 VCP-only。
- 两个不同 VCP 不会因空杯柄字段而错误去重。
- 杯柄现有序列化结果保持兼容。

#### 执行顺序要求

必须先完成 BUG-006 的统一策略引擎设计，再执行 BUG-005 的历史回测迁移。

### PLAN-005：数据源顺序修复不能破坏 source-busy 语义

#### 原方案问题

原方案要求按配置顺序迭代，并“记录失败源，不再重试”。

但必须区分：

- 数据源请求真实失败
- 数据源锁暂时被其他线程占用
- 所有源暂时繁忙

锁忙是瞬时状态，不能作为永久失败记录，否则可能跳过本来可用的数据源。

#### 必须修改为

明确三源选择状态：

```text
not_tried
busy
failed
succeeded
```

建议规则：

1. 按配置顺序尝试尚未尝试的源。
2. 锁忙标记为 `busy`，不得标记为真实失败。
3. 当前轮有其他空闲源时可以继续回退。
4. 所有配置源均锁忙时返回 `data source busy`，由扫描层 requeue。
5. 只有真实请求失败才进入 `failed`。
6. 主源使用 `retry_attempts`，备用源使用 `fallback_attempts`。
7. 保留 `try...finally` 释放锁。

#### 必须增加测试

- 百度锁忙时能够按定义尝试新浪。
- 三个源全部锁忙时返回 `data source busy`。
- 全部锁忙时扫描任务会 requeue。
- source-busy 重试预算耗尽后才标记失败。
- 请求失败源不会在同一轮重复尝试。
- 备用源严格使用 `fallback_attempts`。
- 全部 fresh 源失败时不得使用旧缓存扫描。

#### 必须增加测试

- 不会请求 `sz000001` 或 `szsh000001`。

### PLAN-007：Pivot 下方阈值必须配置化并明确状态

#### 原方案问题

原方案直接使用：

```python
current >= pivot * 0.85
```

15% 下限没有策略文档或回测依据。

同时，“远低于 Pivot → WAIT_ENTRY 或 WAIT_STABLE”没有明确唯一状态。

#### 必须修改为

增加明确配置，例如：

```yaml
decision:
  near_pivot_below_pct: 10
  chase_threshold_pct: 5
```

定义：

```python
near_pivot = (
    pivot > 0
    and current >= pivot * (1 - near_pivot_below_pct / 100)
    and current <= pivot * (1 + chase_threshold_pct / 100)
)
```

远低于 Pivot 且不在低吸区时，明确返回 `WAIT_ENTRY`。

#### 必须增加测试

- 当前价位于 Pivot 下方允许范围内时 `near_pivot=True`。
- 当前价低于允许范围时 `near_pivot=False`。
- 当前价远低于 Pivot 时返回 `WAIT_ENTRY`。
- 当前价高于 Pivot 追高阈值时返回拒绝。

### PLAN-008：复权问题必须先形成确定方案

#### 原方案问题

原方案仍使用：

```text
验证是否默认前复权
如果不是则添加参数
```

这属于调查任务，不是可直接执行的修复方案。

同时没有处理已有 SQLite 缓存可能混合不同复权口径的问题。

#### 必须修改为

实施前先完成一次独立调查并记录结果：

| 数据源 | 当前返回口径 | 是否支持前复权 | 策略扫描是否允许使用 |
|---|---|---|---|
| baidu | 待确认 | 待确认 | 待确认 |
| sina | 待确认 | 待确认 | 待确认 |
| tencent | qfq | 是 | 是 |

调查完成后再选定方案：

1. 三个源统一输出前复权数据；或
2. 无法保证前复权的数据源退出策略扫描。

同时必须定义缓存处理策略：

- 增加复权口径元数据；或
- 修复上线后清空并重建旧 OHLC 缓存；或
- 使用版本字段使旧缓存失效。

不得继续按日期直接合并无法确认复权口径的数据。

#### 必须增加测试

- 使用固定样本验证归一化后的跨源价格连续性。
- 不使用实时外部网络请求作为单元测试通过条件。
- 不同复权口径数据不得被静默合并。
- 旧缓存失效或迁移策略有测试覆盖。
