# 策略3盈亏比优先极致优化设计

## 1. 背景与目标

当前策略3「强势回踩二次启动」已经具备本地 DB 回测闭环，最终可信任务 `s3bt-final-risk04-20260630-132932` 的结果如下：

- 评估股票：4898
- 评估日：1,460,255
- 合并机会：30
- 胜率：36.67%
- 平均盈利：+15.51%
- 平均亏损：-3.43%
- 盈亏比：约 4.52:1
- 单笔期望：+3.51%

这说明策略3当前不是高胜率策略，而是高赔率、低频、靠少数大赚覆盖多次小亏的策略。下一阶段优化目标不是扩大候选数量，也不是强行提高胜率，而是继续围绕盈亏比、Profit Factor、单笔期望和最大回撤风险，把策略3打磨成“少而精”的高赔率交易机会识别器。

核心目标：

1. 在策略3现有逻辑基础上继续优化，不新建独立策略。
2. 不以候选数量为目标，允许候选继续少。
3. 胜率不作为第一目标，只要求亏损可控、盈利空间足够、期望值为正。
4. 所有调参必须先由本地历史回测和分组统计支持。
5. 不修改策略1、策略2核心逻辑。
6. 不引入新的外部行情源；指数数据优先通过现有 `scanner/index_source.py` 能力接入，回测必须可使用本地/缓存指数数据或显式降级。

## 2. 当前问题判断

### 2.1 最终机会数只有 30

30 个机会对“高频扫描候选”来说偏少，但对“高赔率低频交易机会”可以接受。真正的问题不是数量少，而是样本稳定性不足：

- 机会明显集中在 `2025-12`，该月 14 个机会、7 个 TARGET，是主要收益来源。
- `2025-03`、`2025-05` 表现明显较差。
- 如果继续按当前 30 个样本微调参数，容易过拟合某个月份。

下一步不能简单放宽阈值增加样本，也不能按月份过滤。正确方向是引入更合理的市场/板块环境解释变量，判断高赔率机会在哪类市场环境中更可靠。

### 2.2 当前大盘环境不够精细

当前策略3回测使用本地等权市场代理；扫描路径主要使用单一配置指数，默认 `sh000001` 上证指数。

这会带来偏差：

- 沪市主板股票用上证指数较合理。
- 深市主板/中小板股票更适合看深证成指。
- 创业板股票更适合看创业板指。
- 科创板股票更适合看科创板相关指数；如果指数源不可用，应明确降级策略。

下一阶段应按股票代码映射对应指数，回测和扫描都能知道“这只股票对应的大盘环境是什么”。

### 2.3 当前指标偏“入选过滤”，缺少“赔率解释”

策略3已有风险比、RR1、目标价、止损价，但回测报告还不够突出高赔率交易需要的关键指标：

- 平均盈利 / 平均亏损
- Profit Factor
- 单笔期望
- 最大连续止损
- 目标空间分布
- 持仓天数分布
- 目标是否来自真实压力位而非过度乐观估计

下一阶段应把这些作为核心验收指标，而不是只看胜率、总分或候选数量。

## 3. 优化总原则

本阶段使用“盈亏比优先”原则：

1. 优先保留亏损小、盈利空间大的机会。
2. 允许胜率低于 50%，但平均盈利必须显著大于平均亏损。
3. 允许候选数量少，但不能少到无法继续验证；若最终机会低于 20，需要标记为样本不足，而不是宣称成功。
4. 不为了提高胜率而引入过晚确认，避免把赔率吃掉。
5. 不为了提高候选数而放宽止损空间、降低 RR 要求或允许弱势市场环境。
6. 不按单个月份或少数个股做硬编码过滤。

推荐核心验收指标：

| 指标 | 目标 |
|---|---:|
| 平均盈利 / 平均亏损 | >= 4.0 |
| Profit Factor | >= 2.0 |
| 单笔期望 | > 0 |
| 20 日期末平均收益 | > 0 |
| 机会数 | >= 20，低于 20 只能标记观察 |
| 评估异常 | 0 |
| 可信状态 | TRUSTED_BASELINE |

## 4. 方案设计

### 4.1 股票到指数的映射

新增策略3专用指数映射能力，不影响策略1、策略2。

建议映射：

| 股票代码前缀 | 指数名称 | 新浪符号 |
|---|---|---|
| `600` / `601` / `603` / `605` | 上证指数 | `sh000001` |
| `000` / `001` / `002` / `003` | 深证成指 | `sz399001` |
| `300` / `301` | 创业板指 | `sz399006` |
| `688` | 科创50 或科创板代理 | 优先尝试 `sh000688` |

若 `sh000688` 数据源不可用，科创板默认降级为 `sh000001`，但必须在回测信号快照中记录 `market_index_fallback=true` 和 fallback 原因。

### 4.2 指数数据接入

扫描路径：

- 启动策略3扫描时，按配置预加载所需指数集合，而不是每只股票单独请求。
- 将股票代码映射到指数 symbol，再把对应指数窗口传给 `StrongPullbackSecondBreakoutEngine.evaluate_at()`。
- 如果指数获取失败，不应阻断扫描，但必须记录 fallback reason。

回测路径：

- 优先支持从本地缓存表读取指数日线，保证回测不访问外部网络。
- 如果当前项目尚无指数缓存表，第一阶段可以在回测任务启动时接受 `market_data_by_symbol` 注入，测试用本地构造数据。
- 不允许回测过程中调用新浪、腾讯、百度等外部源。
- 如果没有真实指数数据，则继续使用本地等权市场代理，但必须在任务 summary 中标明 `market_data_mode = local_equal_weight_proxy`，不能和真实指数回测混淆。

### 4.3 市场环境特征

每个信号快照新增以下市场环境字段：

- `market_index_symbol`
- `market_index_name`
- `market_return_20`
- `market_return_60`
- `market_ma20`
- `market_ma60`
- `market_above_ma20`
- `market_above_ma60`
- `market_volatility_20`
- `market_drawdown_60`
- `market_data_mode`
- `market_index_fallback`
- `market_index_fallback_reason`

这些字段第一阶段只用于回测分组统计，不直接硬过滤，避免在样本不足时过早收紧规则。

### 4.4 盈亏比核心统计

回测 summary 需要新增或强化以下统计：

- `avg_win`
- `avg_loss`
- `payoff_ratio`
- `profit_factor`
- `expectancy`
- `max_consecutive_losses`
- `target_hit_rate`
- `stop_hit_rate`
- `avg_holding_days`
- `median_holding_days`

分组统计至少覆盖：

- 对应指数 symbol
- 市场 20 日收益区间
- 市场 60 日收益区间
- 市场是否在 MA20 / MA60 之上
- 风险比区间
- RR1 区间
- trade_state
- 月份

重点不是找最高胜率分组，而是找 Profit Factor、payoff ratio 和 expectancy 更稳定的分组。

### 4.5 高赔率过滤层

在完成指数分组回测前，不直接新增硬过滤。

只有当回测证明某些市场环境显著劣化赔率时，才允许加入策略3自身过滤项，例如：

- 对应指数 60 日收益过强，已经进入追高环境，导致后续赔率下降。
- 对应指数 60 日收益为负且价格在 MA60 下方，个股低吸失败率显著升高。
- 对应指数波动率过高，止损触发明显增多。

任何正式过滤必须满足：

1. 样本数不少于 20；若不足 20，只能作为观察结论。
2. 过滤后 Profit Factor 提高。
3. 过滤后单笔期望不下降。
4. 过滤后不是单个月份贡献绝大多数收益。
5. 不降低已有 RR 和风险控制标准。

## 5. 实施边界

允许修改：

- `strategy3/` 内部指标、配置、回测、扫描辅助逻辑。
- `scanner/index_source.py` 的指数获取复用能力，但不得影响策略1现有默认行为。
- `scanner/db.py` 中策略3回测表的兼容字段扩展。
- `tests/test_strategy3_*`。
- 策略3回测报告与设计文档。

禁止修改：

- 策略1核心判断入口和评分逻辑。
- 策略2核心判断入口、评分、回测逻辑。
- 生产日线数据源链。
- 已有策略3候选旧字段，不能删除或改名。
- 回测执行模型，仍使用 `NEXT_OPEN`。

## 6. 推荐开发步骤

### Phase 1：指数映射与快照追踪

1. 新增策略3专用 `resolve_strategy3_market_index_symbol(code)`。
2. 为沪市、深市、创业板、科创板添加单元测试。
3. 在指标结果或信号快照中记录对应指数 symbol/name。
4. 扫描和回测均保留旧的无指数 fallback 行为。

### Phase 2：回测支持多指数市场数据

1. 回测服务支持 `market_data_by_symbol`。
2. 每只股票按代码选择对应指数窗口。
3. 防未来函数：每个评估日只能看到 `<= evaluation_date` 的指数数据。
4. 若指数数据缺失，记录 fallback，不抛异常。

### Phase 3：盈亏比统计增强

1. 回测 summary 增加 `avg_win`、`avg_loss`、`payoff_ratio`、`profit_factor`、`expectancy`、`max_consecutive_losses`。
2. 增加按指数和市场环境分组的统计。
3. 生成新的回测报告，明确哪些环境提高或降低赔率。

### Phase 4：数据驱动规则调整

1. 只基于 Phase 3 的统计结果调整策略3参数或新增过滤。
2. 每个调参点必须先写测试。
3. 跑完整本地 DB 回测验证。
4. 只接受提高 Profit Factor / expectancy 且不明显过拟合月份的规则。

## 7. 测试计划

必须新增或覆盖：

1. 股票代码到指数 symbol 映射测试。
2. 科创板指数不可用时 fallback 测试。
3. 回测不访问外部行情源测试。
4. 多指数数据按股票正确选择测试。
5. 指数窗口不包含未来日期测试。
6. 信号快照包含 market index 字段测试。
7. summary 包含盈亏比核心指标测试。
8. 按指数分组统计测试。
9. 策略1、策略2独立性测试继续通过。
10. 完整后端回归测试继续通过。

推荐命令：

```bash
python -m pytest tests/test_strategy3_engine.py tests/test_strategy3_backtester.py tests/test_strategy3_backtest_db.py tests/test_strategy3_backtest_service.py tests/test_strategy3_db_api.py -q
python -m pytest tests/ -q --ignore=tests/test_akshare_hist.py --ignore=tests/test_tushare_hist.py
python -m compileall scanner strategy2 strategy3 server.py -q
```

## 8. 验收标准

开发完成后必须交付：

1. 一轮使用多指数或明确 fallback 模式的策略3本地 DB 回测任务。
2. 回测任务状态必须为 `TRUSTED_BASELINE`。
3. 回测任务失败股票数和评估异常数必须为 0。
4. 报告必须列出：
   - 总机会数
   - 胜率
   - 平均盈利
   - 平均亏损
   - 盈亏比
   - Profit Factor
   - 单笔期望
   - 最大连续止损
   - 月度分布
   - 指数分组表现
5. 若最终机会数低于 20，不得宣称策略已优化成功，只能标记为样本不足。
6. 若收益主要集中在单个月份，不得继续按该结果硬调参数。
7. 不允许为了提高候选数而放宽风险比或 RR 要求。
